## 为什么需要 GRPO？

标准 RLHF 的 PPO 训练需要 **4 个模型**同时驻留 GPU：

| 模型 | 用途 | 大小 |
|------|------|------|
| Policy Model | 当前策略 | ~模型本体 |
| Reference Model | KL 锚点 | ~模型本体 |
| Reward Model | 偏好打分 | 较小 |
| **Critic (Value) Model** | **优势估计** | **~模型本体** |

Critic Model 与 Policy Model 等大——在 671B 规模上，这是数百 GB 的显存浪费。

## GRPO 的核心思想

> **抛弃 Critic！** 对同一个 prompt 采样 G 个回答，用组内均值和标准差归一化 reward 作为 advantage。

```python
# PPO: 需要 Critic
A = reward + gamma * V(next) - V(current)  # V 来自 Critic

# GRPO: 组内归一化
A_i = (reward_i - mean(rewards)) / std(rewards)  # 不需要 Critic！
```

## 从 7B 到 671B

| 论文 | 模型 | 任务 | 涌现 |
|------|------|------|------|
| DeepSeekMath (2402.03300) | 7B | 数学推理 | GRPO 首次提出 |
| DeepSeek-R1 (2501.12948) | 671B | 通用推理 | **Aha Moment** |

GRPO 在 7B 上验证了可行性，在 671B 上展现了 emergent reasoning——模型在没有人类推理数据的情况下，自发学会了 **self-reflection**、**verification**、**alternative approaches**。

## 面试要点

- GRPO 比 PPO 省 ~25% 显存（去掉 Critic）
- Group size G=4 是经验平衡点
- 缺陷：如果 group 内所有回答都正确 → advantage 全 0 → 梯度消失
